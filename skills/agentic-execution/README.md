# Agentic Execution Planner

Generate execution plans that function as live orchestration runbooks for
dispatching specialist multi-agent swarms. Traditional execution is ad-hoc --
a developer reads the implementation plan and manually spawns agents. Agentic
execution plans are structured runtime playbooks: they specify exactly how to
dispatch each wave of specialist agents, what to monitor during execution, when
to gate and merge, how to recover from failures in real-time, and when to
adaptively re-plan.

This is the deeply agent-specific layer of the pipeline. All decisions about
orchestration model, specialist agent roles, model selection, dispatch commands,
budget controls, and runtime monitoring live here.

## Position in the Pipeline

```
PRD (agentic-prd)              -- what to build and why
Architecture (agentic-arch)    -- how it decomposes structurally
Implementation (agentic-impl)  -- tasks, prompts, verification gates
Execution (this skill)         -- live dispatch, monitoring, completion
```

This skill consumes the implementation plan and makes it executable. The
implementation plan answers "what tasks, in what order, with what prompts?"
This skill answers "how do I actually run this right now, monitor it live,
handle failures as they happen, and drive it to completion?"

## Prerequisites

This skill expects an **Implementation Plan** (generated by the
`agentic-implementation` skill or equivalent).

The implementation plan's output contract provides everything needed:
- Task breakdown with IDs, dependencies, files, sizing, success criteria
- Execution schedule with waves, parallel lanes, and critical path
- Spawn prompts for every agent
- Verification gates at all four levels
- Merge strategy with ordering and procedures
- Recovery procedures for each failure type

If the implementation plan is missing, advise the user to generate it first.
If a PRD and architecture plan exist, reference them for acceptance criteria
and module context, but the implementation plan is the primary input.

## Workflow

### 1. Consume the Implementation Plan

Read the implementation plan and extract its output contract:

- **Task Breakdown** -- IDs, dependencies, files, sizing, success criteria
- **Execution Schedule** -- waves, parallel lanes, critical path
- **Spawn Prompts** -- complete agent prompts per task
- **Verification Gates** -- gate definitions at all four levels
- **Merge Strategy** -- merge ordering and procedures
- **Recovery Procedures** -- failure handling per failure type

**Quick-validate:**
- Every task has a spawn prompt
- Every wave has entry/exit gates with executable verification commands
- Merge ordering is consistent with the DAG
- Budget estimates are derivable (task count x sizing x model costs)

If gaps exist, flag them. Execution plans built on incomplete implementation
plans fail at runtime.

### 2. Select the Orchestration Runtime & Assign Agents

This is where all agent-specific decisions are made. Determine HOW agents will
be dispatched based on project characteristics and the implementation plan's
structure.

See [references/orchestration-runtimes.md](references/orchestration-runtimes.md)
for runtime selection criteria and configuration.
See [references/agent-assignment.md](references/agent-assignment.md) for
agent assignment strategies and orchestration patterns.

**Three runtime options:**

| Runtime | Best For | Dispatch Method |
|---------|----------|----------------|
| **Subagents with Worktrees** | Independent tasks, clear file ownership | `Agent` tool with `isolation: worktree` |
| **Agent Teams** | Complex coordination, inter-agent communication | `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` |
| **Manual Worktree Sessions** | Maximum control, mixed models, large projects | `git worktree add` + separate `claude` sessions |

**Selection factors:**
- Inter-agent communication needed? -> Agent Teams
- >15 tasks with complex dependencies? -> Agent Teams or Manual Worktrees
- Simple parallel work with no cross-talk? -> Subagents
- Need different models per agent? -> Manual Worktrees
- Budget-sensitive? -> Subagents (lowest overhead)
- Need automated quality enforcement? -> Agent Teams (TeammateIdle/TaskCompleted hooks)

**Agent assignment decisions:**
- Map implementation plan lanes to concrete agent roles
- Select model tier per role (Opus for lead/foundation/critical path, Sonnet
  for parallel feature work, Haiku for exploration/research)
- Determine plan approval requirements for high-risk modules (Agent Teams
  support requiring teammates to plan in read-only mode before implementing;
  the lead reviews and approves/rejects plans with feedback)
- Right-size agent count: 3-5 parallel coding agents is the sweet spot
  (Google/MIT research shows performance degrades beyond this)
- Target 5-6 tasks per teammate to keep agents productive without excessive
  context switching

### 3. Build the Wave Dispatch Plan

Transform the implementation plan's execution schedule into concrete dispatch
commands. Each wave becomes a dispatchable unit with pre-conditions, commands,
monitoring hooks, and exit gates.

See [references/wave-dispatch.md](references/wave-dispatch.md) for dispatch
templates and patterns.

**Every wave dispatch must specify:**
- **Wave ID** and human-readable name
- **Pre-conditions** (gates that must have passed, artifacts that must exist)
- **Dispatch commands** (exact CLI commands or Agent tool calls to spawn each agent)
- **Agent configurations** (model, isolation mode, permission mode, custom agent file)
- **Monitoring hooks** (what to watch during execution, how often to check)
- **Exit gate** (verification commands that must pass before the next wave)
- **Timeout** (max wall-clock time before escalation)
- **Budget ceiling** (max tokens/cost for this wave)

### 4. Configure Specialist Agent Definitions

Create `.claude/agents/` definition files for each specialist role needed.
Specialist agents outperform generalists because they have narrow, focused
system prompts with restricted scope.

See [references/specialist-agents.md](references/specialist-agents.md) for
agent definition templates and role patterns.

**Standard specialist roles:**

| Specialist | Focus | Model | When Used |
|-----------|-------|-------|-----------|
| Foundation Builder | Shared types, schemas, utils | opus | Wave 1 |
| Feature Implementer | Single module implementation | sonnet | Wave 2+ |
| Integration Wirer | Route composition, middleware, E2E | sonnet/opus | Final wave |
| Test Hardener | Test coverage gaps, edge cases | sonnet | Post-integration |
| Security Reviewer | SAST patterns, auth flows, input validation | sonnet | Post-integration |
| Performance Auditor | Bundle size, query optimization, caching | sonnet | Post-integration |
| Merge Conductor | Sequential merge, conflict resolution | opus | Wave transitions |
| Recovery Agent | Diagnose failures, fix broken modules | opus | On failure |

### 5. Generate the Execution Runbook

The execution runbook is the step-by-step playbook the operator follows to
execute the full implementation plan with agent swarms. It transforms the
abstract implementation plan into a concrete sequence of actions.

See [references/execution-runbook.md](references/execution-runbook.md) for
the complete runbook template.

**The runbook must include:**
- **Pre-flight checklist** (environment setup, git state, dependencies)
- **Wave-by-wave dispatch instructions** (exact commands, copy-paste ready)
- **Live monitoring dashboard** (what to watch, how to check progress)
- **Gate verification scripts** (executable bash scripts for each gate)
- **Merge sequence commands** (exact git commands, derived from the
  implementation plan's merge strategy)
- **Completion criteria** (how to know when the execution is done)

### 6. Define the Adaptive Re-Planning Strategy

Live execution rarely goes exactly as planned. The execution plan must define
triggers and procedures for adaptive re-planning without halting the swarm.

See [references/adaptive-replanning.md](references/adaptive-replanning.md) for
re-planning triggers and procedures.

**Adaptive re-planning covers:**
- **Task splitting** -- when a task is too large, split it mid-execution
- **Agent replacement** -- when an agent fails, respawn with updated context
- **Wave restructuring** -- when dependencies shift, reorganize remaining waves
- **Scope reduction** -- when budget is exceeded, identify the minimum viable subset
- **Contract evolution** -- when foundation changes are needed mid-execution
- **Cascading failure circuit breaker** -- halt conditions and recovery protocol

### 7. Build the Budget & Resource Plan

Multi-agent execution is expensive. The execution plan must include explicit
budget controls, cost projections, and early-stopping triggers.

See [references/budget-controls.md](references/budget-controls.md) for
budgeting methodology.

**Budget plan includes:**
- **Per-wave cost projection** (agent count x model cost x estimated time)
- **Per-agent token budget** (ceiling per agent session)
- **Total execution budget** (hard ceiling with early-stopping trigger)
- **Model tier optimization** (which agents can use cheaper models)
- **Cost monitoring commands** (how to check current spend during execution)
- **Early-stopping criteria** (when to halt and re-assess vs. continue)

### 8. Write the Execution Plan

Generate the execution plan using the template in the reference files.

See [references/execution-template.md](references/execution-template.md)
for the complete template.

Every execution plan must include:
- **Execution Overview** with runtime, wave count, budget, and timeline
- **Pre-Flight Checklist** for environment and state verification
- **Specialist Agent Definitions** with .claude/agents/ file content
- **Wave Dispatch Plan** with exact commands for each wave
- **Live Monitoring Dashboard** with progress tracking setup
- **Gate Verification Scripts** for every checkpoint
- **Merge Sequence** with exact commands (implementing the implementation
  plan's merge strategy as copy-paste-ready commands)
- **Adaptive Re-Planning Strategy** with triggers and recovery circuits
- **Budget & Resource Plan** with cost projections and early-stopping
- **Completion Checklist** mapping back to PRD acceptance criteria
- **Post-Execution Retrospective Template** for process improvement

### 9. Deliver

Write the execution plan to a file (default: `EXECUTION.md` in the project
root, or as the user specifies).

Also generate:
- `.claude/agents/` definition files for each specialist role
- `scripts/` directory with gate verification scripts
- `scripts/dispatch-wave-N.sh` for each wave (if using Manual Worktrees)

Provide the user with:
- A summary of wave count, agent count, and estimated total execution time
- The critical path and any bottleneck warnings
- The total budget projection
- Clear instructions for how to begin execution (exact first command to run)
- Any risks or open questions that need resolution before launching

## Research-Backed Execution Thresholds

These empirically validated thresholds should guide execution plan generation:

- **45% single-agent threshold** (Google/MIT, Dec 2025): If a single agent
  exceeds 45% success on a non-decomposable task, adding more agents DEGRADES
  performance. Multi-agent only helps for parallelizable, decomposable work.
- **35-minute degradation cliff** (Zylos Research, 2026): Agent performance
  degrades after 35 minutes of continuous operation. Doubling task duration
  quadruples the failure rate. Time-box all agent sessions.
- **79% specification failure rate** (MAST/UC Berkeley, 2025): 79% of
  multi-agent failures originate from specification and coordination issues,
  NOT from model limitations. The upstream pipeline's quality determines success.
- **17x error amplification** (Google/MIT): Uncoordinated "bag of agents"
  architectures amplify errors 17.2x. Centralized orchestration reduces this
  to 4.4x. Always use structured wave dispatch with gates.
- **Context rot starts immediately** (Chroma Research, 2026): Performance
  degrades at every context length increment, not just near the limit.
  Effective context is ~200K tokens even with 1M windows. Keep spawn prompts
  focused (ETH Zurich: context files >300 lines reduce success by ~3% and
  increase costs by >20%).
- **90.2% improvement** (Anthropic): Opus lead + Sonnet workers outperformed
  single-agent Opus by 90.2%. Model stratification is the highest-leverage
  optimization.
- **15x token cost** (Anthropic): Multi-agent systems use ~15x more tokens
  than single-agent. Budget controls are essential, not optional.

## Key Principles

1. **Execution plans are runtime playbooks** -- they specify exactly what
   commands to run, what to monitor, and what to do when things go wrong;
   an operator should be able to follow the plan step-by-step without
   improvisation
2. **Specialist agents outperform generalists** -- narrow scope, focused
   prompts, and restricted tools produce better output than "do everything"
   agents; define explicit specialist roles for every wave
3. **Budget is a first-class constraint** -- multi-agent execution costs
   15-50x more than single-agent; every wave has a token ceiling, every
   agent has a cost budget, and early-stopping triggers prevent runaway spend
4. **Adaptive re-planning is expected** -- no execution goes perfectly;
   build recovery circuits into every wave so failures are handled
   automatically rather than requiring manual intervention
5. **Gates are non-negotiable** -- no wave advances without its gate
   passing; no merge happens without test-after-each; gates catch errors
   when they are cheapest to fix
6. **Monitor continuously, not after the fact** -- live progress tracking
   detects problems in minutes instead of discovering them at integration
   time; define monitoring hooks for every wave
7. **Dispatch is copy-paste ready** -- every command in the execution plan
   should be directly executable; no "adapt this to your environment"
   placeholders; the operator copies, pastes, and runs
8. **Time-box every agent** -- no agent should run longer than 30-35
   minutes without a checkpoint commit; context degradation is empirically
   proven to start early and accelerate; fresh spawns are cheaper than
   degraded output (Zylos Research, Chroma Research)
9. **Model stratification saves money** -- Opus for lead/foundation/
   critical path; Sonnet for parallel feature work; Haiku for exploration
   and research; Anthropic confirms "Opus lead with Sonnet workers
   outperformed single-agent Opus by 90.2%"
10. **Know when NOT to swarm** -- if <10 tasks, tightly coupled modules,
    or a single agent can complete the work in <30 min, single-agent
    execution outperforms multi-agent (Google/MIT 45% threshold)
