# Agentic Implementation Planner

Generate implementation plans that function as executable schedules for multi-agent
coding swarms. Traditional implementation plans optimize for human sprint planning;
agentic implementation plans optimize for agent-executable task granularity,
parallel dispatch, TDD verification gates, and conflict-free merge convergence.

The implementation plan is the agent-aware decomposition layer: it takes the
architecture plan (what to build and how it decomposes structurally) and produces
the exact tasks, prompts, and verification gates each agent will execute. It is
agent-aware (tasks are sized for context windows, scoped to avoid file conflicts,
ordered for parallel execution) but not agent-specific (it does not choose
orchestration runtimes or define specialist agent configurations -- that belongs
to the downstream execution skill).

## Key Principles

1. **Tasks are the unit of work** -- every agent receives one task at a time
   with clear inputs, outputs, and verification criteria; never hand an agent
   an entire module without decomposition
2. **Solvability, completeness, non-redundancy** -- every task must be
   independently solvable, the full set must cover all requirements, and no
   two tasks should overlap (AOP framework, ICLR 2025)
3. **Test first, implement second** -- every task begins with writing tests that
   define the expected behavior; implementation follows until tests pass
4. **Isolate everything** -- every agent works on its own files; no shared
   mutable state during execution
5. **Gate every transition** -- no task advances without passing its verification
   gate; no wave starts without the prior wave's gate passing
6. **Spawn prompts are contracts** -- the quality of the spawn prompt determines
   the quality of the output; invest heavily in precise, complete prompts
7. **Plan for failure** -- every task has a recovery procedure; every merge has
   a rollback path; every agent can be respawned from scratch
8. **Foundation is sacred** -- Tier 0 tasks build the shared types, schemas,
   and utilities that everything else depends on; rush the foundation and
   every downstream task will fail
9. **Ship in waves** -- each execution wave should leave the codebase in a
   runnable, testable state; no wave should break the build

## Position in the Pipeline

```
PRD (agentic-prd)              -- what to build and why
Architecture (agentic-arch)    -- how it decomposes structurally
Implementation (this skill)    -- tasks, prompts, verification gates
Execution                      -- live dispatch, monitoring, completion
```

This skill consumes the architecture plan (and references the PRD for acceptance
criteria). It produces an implementation plan that the execution skill transforms
into runtime dispatch commands.

## Prerequisites

This skill expects one or both of the following inputs:
- A **PRD** (generated by the `agentic-prd` skill or equivalent)
- An **Architecture Plan** (generated by the `agentic-architecture` skill or equivalent)

If neither exists, advise the user to generate them first. The implementation plan
cannot be built from vague requirements -- it needs the module boundaries, dependency
DAG, and interface contracts from the architecture plan.

## Workflow

### 1. Consume the Architecture Plan

Read the architecture plan and extract its output contract:

- **Module Boundary Map** -- every module, its files, its single owner
- **Dependency DAG** -- tiers, critical path, parallelization width
- **Interface Contracts** -- type signatures, API schemas, mock data
- **Integration Checkpoints** -- where parallel work must sync
- **CLAUDE.md Recommendations** -- conventions, constraints, frozen foundations

From the PRD, extract:
- Acceptance criteria (machine-verifiable) for mapping to tasks
- Protection patterns (DO NOT CHANGE sections, Always/Ask First/Never)
- Non-goals and explicit constraints

**Validate the essentials:**
- Every module has a clear file manifest with no overlaps
- Interface contracts exist for every cross-module boundary
- The DAG has no cycles and the critical path is identified

If gaps exist, flag them and ask the user. An implementation plan built on an
incomplete architecture will fail at execution time.

### 2. Decompose Modules Into Tasks

Convert each architecture module into a sequence of concrete, agent-executable
tasks. Task decomposition quality is the single biggest determinant of success.

See [references/task-decomposition.md](references/task-decomposition.md) for the
complete decomposition methodology.

**Apply the AOP decomposition principles** (Agent-Oriented Planning, ICLR 2025):

- **Solvability**: Each task must be independently completable by a single agent
  within one context window (~35 minutes of work, per Zylos Research degradation
  findings). If an agent cannot solve the task alone, it is too large or has
  hidden dependencies.
- **Completeness**: The full set of tasks must cover every requirement in the
  architecture plan and PRD. No requirement should be orphaned.
- **Non-redundancy**: Each task should be unique and necessary. Two tasks should
  never produce overlapping outputs or modify the same files.

**Every task must specify:**
- A unique task ID (e.g., `T0.1`, `T1.3`, `T2.5`)
- The parent module and DAG tier
- A clear, one-sentence objective
- Input dependencies (task IDs that must complete first)
- Files to create or modify (exact paths)
- Files to read but not modify (for context)
- Interface contracts to implement or consume
- Test requirements (what tests to write first, what tests must pass)
- Success criteria (machine-verifiable, derived from PRD acceptance criteria)
- Estimated complexity (S/M/L -- correlating to ~5/10/15 min agent work)

**Task sizing rules:**
- Target 5-15 minutes of agent work per task
- Each task must produce a testable, committable unit of work
- No task should touch files owned by another module
- Foundation tasks (Tier 0) should be smaller and more numerous -- they gate everything
- Feature tasks can be larger since they run in parallel with less risk

### 3. Build the Execution Schedule

Convert the task list into a time-sequenced execution schedule that maps tasks
to parallel lanes across execution waves.

See [references/execution-scheduling.md](references/execution-scheduling.md) for
scheduling methodology.

**The schedule must define:**
- **Execution waves** (groups of tasks that run in parallel)
- **Parallel lane assignments** (which lane runs which tasks in sequence)
- **Synchronization gates** (points where all lanes must complete before advancing)
- **The critical path** (the longest sequential chain of tasks)
- **Estimated total execution time** (wall-clock with parallelism factored in)

**Scheduling principles:**
- Maximize wave width (more parallel tasks = faster completion)
- Foundation tasks (Tier 0) execute first -- they are the shared dependency and
  must be rock-solid before anything else starts
- Feature tasks dispatch in parallel after foundation completes
- Integration tasks run after all feature tasks in their dependency chain complete
- Stay within 3-5 parallel lanes (performance degrades beyond this threshold,
  per Google/MIT research)

### 4. Generate Spawn Prompts

Write the exact prompt each agent will receive when spawned. Spawn prompt quality
is the single biggest lever for agent performance.

See [references/spawn-prompts.md](references/spawn-prompts.md) for prompt templates
and examples.

**Critical context rule:** Spawned agents (subagents, Agent Team teammates,
and manual worktree sessions) do NOT inherit the lead's conversation history.
They start with a fresh context containing only: CLAUDE.md, MCP servers, skills,
and the spawn prompt you provide. Everything the agent needs to succeed must be
in the spawn prompt itself or in files it can read (like ARCHITECTURE.md). Never
reference "what we discussed" or assume shared conversational context.

**Every spawn prompt must include:**
- The agent's role and specific module assignment
- Architecture context (relevant section of the architecture plan)
- Interface contracts the agent must implement or consume (exact type signatures)
- File manifest (files to create, files to read for context)
- Test-first instructions (write tests before implementation)
- Explicit constraints (DO NOT modify files outside your manifest)
- Success criteria (what "done" looks like, with verification commands)
- The CLAUDE.md conventions the agent must follow

**Spawn prompt anti-patterns to avoid:**
- Vague delegation ("implement the auth module") -- specify exact files and contracts
- Missing constraints -- agents will refactor code outside their scope
- No test requirements -- agents skip tests unless explicitly required
- Overly verbose context -- keep prompts focused; agents discover structure well
  (ETH Zurich: context files >300 lines reduce success by ~3%)
- Referencing the lead's conversation ("as we discussed") -- agents have no
  access to the lead's history; embed all necessary context directly

### 5. Define Verification, Merge & Recovery

Design the verification gates, merge strategy, and recovery procedures. These
three concerns are tightly coupled -- a gate failure triggers a merge rollback,
which triggers a recovery procedure.

See [references/verification.md](references/verification.md) for the complete
verification framework and testing patterns.
See [references/merge-integration.md](references/merge-integration.md) for
merge strategy patterns.
See [references/progress-tracking.md](references/progress-tracking.md) for
tracking and recovery patterns.

**Four-level verification lattice:**

| Level | Trigger | Checks |
|-------|---------|--------|
| Task | After every task | Unit tests pass, lint clean, type check clean |
| Module | After all tasks in a module | Contract compliance, export verification, all module tests |
| Wave | At synchronization gates | Integration tests, sequential merge with test-after-each, regression gate |
| Plan | Final integration | Full E2E suite, build verification, PRD acceptance criteria |

**Merge principles:**
- Sequential merge with test-after-each (merge one branch, run all tests, repeat)
- Foundation branch merges first -- it is the base for all feature branches
- Feature branches merge in dependency order (leaves before roots)
- If a merge introduces test failures, revert and fix before proceeding

**Recovery procedures:**
- Task failure: Discard worktree, diagnose root cause, respawn with updated prompt
- Merge conflict: Identify boundary violation, fix ownership, retry merge
- Contract mismatch: Fix non-compliant module or update foundation contract
- Context exhaustion: Split remaining tasks, spawn fresh agent
- Cascading failure: Halt all agents, reassess architecture, update plan

### 6. Write the Implementation Plan

Generate the implementation plan using the template in the reference files.

See [references/implementation-template.md](references/implementation-template.md)
for the complete template.

Every plan must include:
- **Implementation Overview** with total task count, wave count, estimated time
- **Task Breakdown** with every task fully specified
- **Execution Schedule** with wave assignments and parallel lanes
- **Spawn Prompts** for every agent (ready to copy-paste)
- **Verification Gates** at every checkpoint level
- **Merge Strategy** with explicit merge ordering
- **Progress Tracking** setup and recovery procedures
- **Risk Register** with mitigation strategies

### 7. Deliver

Write the implementation plan to a file (default: `IMPLEMENTATION.md` in the
project root, or as the user specifies).

Provide the user with:
- A summary of the task count and execution timeline
- The critical path and estimated total time
- Any risks or open questions that need resolution before execution
- Clear instructions for how to begin execution (which command to run, which
  agent to spawn first)

If an architecture plan and/or PRD exists, update them to reference the
implementation plan.

#### Output Contract

The implementation plan provides the following to the downstream execution
skill. If any of these are missing, the execution plan cannot be built:

| Output | Description | Used By |
|--------|-------------|---------|
| Task Breakdown | IDs, dependencies, files, sizing, success criteria | Wave dispatch, spawn commands |
| Execution Schedule | Waves, parallel lanes, critical path | Wave dispatch plan |
| Spawn Prompts | Complete agent prompts per task | Specialist agent configuration |
| Verification Gates | Gate definitions at all four levels | Gate verification scripts |
| Merge Strategy | Merge ordering and procedures | Merge sequence commands |
| Recovery Procedures | Failure handling per failure type | Adaptive re-planning |
